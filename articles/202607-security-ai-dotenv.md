## はじめに

`.env` ファイルを平文のままローカル環境やサーバーに保存していませんか？

AI支援ツール（コード補完やAIエージェントなど）が日常的に使われるようになった現在、開発環境における機密情報（APIキーやデータベースのパスワードなど）の取り扱いは、これまでにない大きなセキュリティリスクに直面しています。AIエージェントにプロジェクトのコンテキストを丸ごと読み込ませる機会が増えたぶん、単一の対策に頼るのではなく、複数の層で機密情報を守る「多層防御」の発想が重要になってきました。

従来のように `.env` ファイルを平文のテキストファイルとしてディスクに保存・運用している場合、以下のような理由から重大なセキュリティインシデントに繋がる危険性があります。

## 対象者

* AIが組み込まれたエディタや外部のAIエージェントを日常的に使っている方
* `.env` ファイルを平文のままリポジトリやサーバーに置いている方
* 環境変数の管理を「動けばいい」から一歩進めたいと考えている方

## なぜ、平文の `.env` は危険なのか？

* AIによる意図しない読み込み。エディタに組み込まれたAIや外部のAIアシスタントにプロジェクトのコンテキストを読み込ませた際、`.env` の平文データがそのままプロンプトのコンテキストとして送信されてしまうリスクです。
* ログやエラーハンドリングへの混入。デバッグ時などにフレームワークが自動出力するログやエラー画面に、環境変数の平文がそのまま出力・混入してしまうリスクです。
* 不正アクセス時の容易な窃取。サーバーやコンテナへの不正アクセスを受けた際、ディスク上に平文で存在する `.env` ファイルが一網打尽に抜き取られてしまうリスクです。

:::message alert
生成AIサービス側の学習ポリシーがどうであれ、`.env` の中身を丸ごとプロンプトやコンテキストに含めてしまう運用そのものがリスクです。「学習に使われないから安全」ではなく、「機密情報がAIの目に触れる経路自体を作らない」ことが本質的な対策になります。
:::

この記事では、平文管理から脱却する最初の一歩として `dotenvx` による暗号化の手順を解説したうえで、それでもなお残るリスクと、さらに一歩進んだ考え方（機密情報をディスクに残さない運用）を紹介します。

## dotenvx とは

`dotenvx` は、Node.jsの定番ライブラリ `dotenv` の作者自身が手がける次世代版で、`.env` ファイルをリポジトリに置いたまま暗号化して管理できるツールです。既存の `dotenv` の使い方をほぼそのまま踏襲できるため、導入コストが低い点が特徴です。

### dotenvx の暗号化の仕組み

`dotenvx encrypt` を実行すると、公開鍵（`DOTENV_PUBLIC_KEY`）と秘密鍵（`DOTENV_PRIVATE_KEY`）のペアが生成されます。

* `.env` ファイル自体が書き換えられ、先頭に `DOTENV_PUBLIC_KEY` が追加されたうえで、各値が `encrypted:...` という暗号文に置き換わります。
* 復号用の秘密鍵は `.env.keys` という別ファイルに書き出されます。

つまりリポジトリにコミットされるのは暗号化済みの `.env` だけになり、復号キーは別ファイル・別経路で管理する構成になります。

:::message
以前は `.env.vault` という専用ファイルに暗号値をまとめる方式が使われていましたが、公式ドキュメントによれば2024年5月のバージョン以降この方式は非推奨（DEPRECATED）となっています。現行バージョンでは `.env` 自体を直接暗号化する方式が採用されているため、これから導入する場合は本記事で紹介する方式で問題ありません。
:::

### 導入と使い方

#### 1. インストール

```bash
npm install @dotenvx/dotenvx --save-dev
```

#### 2. `.env` の暗号化

以下のコマンドを実行すると、`.env` の内容が暗号化され、復号用の秘密鍵が自動生成されます。

```bash
npx dotenvx encrypt
```

*(※復号鍵を保存した `.env.keys` は Git 管理から除外するため、自動的に `.gitignore` に追加されます)*

暗号化後の `.env` は次のような見た目になります。

```bash
#/-------------------[DOTENV_PUBLIC_KEY]--------------------/
#/            public-key encryption for .env files          /
#/       [how it works](https://dotenvx.com/encryption)      /
#/------------------------------------------------------------/
DOTENV_PUBLIC_KEY="03b3..."

# .env
DB_PASSWORD="encrypted:BCyaHAAvY..."
API_KEY="encrypted:D8gv3jH0..."
```

#### 3. アプリケーションの実行

実行時に `dotenvx run --` を挟むことで、`.env.keys` から秘密鍵を読み取り、メモリ上で復号しながら安全にアプリケーションを起動できます。

```bash
npx dotenvx run -- npm run dev
```

#### 4. 本番環境での鍵の扱い

本番環境では `.env.keys` そのものはデプロイせず、ホスティングサービスの環境変数設定画面などに秘密鍵を直接登録します。環境ごとに鍵を分けたい場合、公式ドキュメントでは `.env.production` のような環境別ファイルに対応させる形で `DOTENV_PRIVATE_KEY_PRODUCTION` という命名規則（`DOTENV_PRIVATE_KEY_${environment}`）が定義されています。この命名規則に沿ったキーが設定されていれば、対応する `.env.${environment}` が実行時に自動で復号される仕組みです。

```bash
DOTENV_PRIVATE_KEY_PRODUCTION="4a650a41...." npx dotenvx run -- node index.js
```

こうすることで、復号キー自体もディスク上のファイルとして残らない運用が可能になります。

## dotenvx暗号化だけでは防げないこと

`dotenvx` によって、リポジトリやディスク上に「静止した状態」で存在する `.env` の中身は暗号文になります。ここまでで、うっかりリポジトリにコミットしてしまった場合や、サーバーへの不正アクセスでファイルごと抜き取られた場合のリスクはかなり下げられます。

ただし、AIエージェントが実際に触れているのは、暗号化された `.env` ファイルそのものではなく、アプリケーションの実行中に復号されてプロセスへ展開された後の環境変数です。ここに、暗号化だけでは埋まらない隙間があります。

:::message alert
AIコーディングエージェントの中には、ファイルの内容を読むだけでなく、ターミナルコマンドを自ら実行できるものがあります。この場合、エージェントが `printenv` や `curl` のようなコマンドを実行すると、その出力（＝復号済みの平文の値）がそのままエージェントへの入力として渡ってしまう可能性があります。実際に、AIエージェントがプロセス環境変数やターミナル出力経由で機密情報を取得できてしまう事例が複数報告されています。「ファイルさえ暗号化しておけば安全」とは言い切れない点に注意してください。
:::

つまり `dotenvx` による暗号化は「ディスク上のファイルを暗号文にする」対策であり、「実行中のプロセスに展開された値をAIエージェントの目から隠す」対策までは提供していません。この2つは別の防御層として捉える必要があります。

## さらに一歩進める：機密情報をディスクに残さない考え方

`dotenvx` で `.env` を暗号化していても、それを使う側の起動スクリプトやフレームワークが「復号結果を一時ファイルに書き出してから読み込む」実装になっていた場合、その一時ファイルが平文のまま残ってしまう可能性があります。暗号化はあくまで「静止しているファイル」を守る対策であり、実行時の取り扱い方まで保証してくれるわけではありません。

そこで意識したいのが、機密情報をディスクへ一切書き出さず、プロセスのメモリ（RAM）上だけで完結させる考え方です。業界で統一された呼称があるわけではありませんが、OWASPのSecrets Management Cheat Sheetが示す「secrets are never written to disk」「ephemeral secrets（揮発性シークレット）」といった原則に近いものです。

### 実装イメージ：シークレットマネージャーからの直接読み込み

AWS Secrets Manager、Google Cloud Secret Manager、HashiCorp Vault などの外部シークレットストアから、起動時にAPI経由で直接データを取得し、ディスクには保存せずプロセスのメモリ上にのみ展開する構成です。Node.jsでAWS Secrets Managerを使う場合、疑似コードにするとおおよそ次のような流れになります。

```javascript
// bootstrap.js（アプリ本体の起動前に実行するエントリーポイント）
const { SecretsManagerClient, GetSecretValueCommand } = require("@aws-sdk/client-secrets-manager");

async function loadSecretsIntoEnv() {
  const client = new SecretsManagerClient({ region: "ap-northeast-1" });
  const command = new GetSecretValueCommand({ SecretId: "prod/myapp/db" });
  const response = await client.send(command);

  const secrets = JSON.parse(response.SecretString);
  for (const [key, value] of Object.entries(secrets)) {
    process.env[key] = value; // ディスクには書き出さず、プロセス環境変数へ直接展開
  }
}

(async () => {
  await loadSecretsIntoEnv();
  require("./server"); // 環境変数がセットされた後にアプリ本体を起動
})();
```

ポイントは、取得したシークレットをファイルに書き出す処理を一切挟まず、`process.env` に直接載せてからアプリ本体を起動することです。IAMロールなどでSecrets Managerへのアクセス権限自体を絞り込んでおけば、万が一コンテナに侵入されても、取得できる情報の範囲を最小化できます。

### 実装時の注意点

:::message alert
* ローカル開発と本番の切り替え。ローカル開発でSecrets Managerまで用意するのはオーバーエンジニアリングになりがちです。ローカルでは `dotenvx` によるメモリ展開にとどめ、本番環境ではシークレットマネージャーとIAMロールなどを連携させる構成が現実的です。
* メモリダンプ対策。クラッシュ時に出力されるコアダンプなどに機密情報が含まれないよう、プロセスの権限管理やクラッシュレポートの設定にも配慮が必要です。
:::

## 3つのアプローチの比較

| | 平文 `.env` | dotenvx暗号化 | シークレットマネージャー連携 |
| --- | --- | --- | --- |
| ディスク上の状態 | 平文で保存 | 暗号文で保存 | 保存しない |
| 導入コスト | - | 低い | 中〜高（外部サービス連携が必要） |
| リポジトリ流出・不正アクセス時の耐性 | ✕ | ○ | ◎ |
| 実行中プロセス経由のAI誤読み込み対策 | ✕ | △（暗号化されるのはファイルのみ。復号後の値はプロセス環境変数としてAIエージェントから見える経路が残る） | ○（IAM権限を絞ればアクセス自体を制限できる） |
| 想定する場面 | （非推奨） | 個人開発・小規模プロジェクトの最低ライン | 機密性の高い本番システム |

## おわりに

AIエージェントに日常的にコードベースを読み込ませるようになってから、`.env` を無防備なまま置いておくことに以前より危うさを感じるようになりました。今回改めて `dotenvx` の公式ドキュメントを読み直してみると、`.env.vault` が2024年5月から非推奨になっていることや、環境別の秘密鍵の命名規則など、細かい仕様が意外と変わっていることにも気づかされました。同時に、暗号化はあくまでディスク上のファイルを守る対策であり、実行中のプロセスに展開された値までは守ってくれないという当たり前の事実にも、調べ直して初めて向き合えた気がします。

まずは平文の `.env` のコミットをやめ、`dotenvx` で暗号化する運用を最低ライン（ベースライン）として徹底する。そのうえで、機密性の高い本番システムについては、シークレットマネージャーとの連携によってディスクに痕跡を残さない運用を検討し、開発スピードと安全性のバランスを取っていく。これが、AI時代における環境変数管理の現実的な落としどころだと思います。この記事が、`.env` の扱いに漠然とした不安を感じている方の参考になれば幸いです。
