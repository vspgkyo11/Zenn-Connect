## はじめに
長年運用されてきたシステムには、時として現代のツールが通用しない「孤島」のような環境が存在します。本記事では、最新のツールが直接動作しない古いOSのサーバーから、手元のクライアントマシンをプロキシとして活用し、大容量データを安全かつ効率的に Google Drive へ同期させる手法を解説します。

## 対象者
- 古い共有サーバーのバックアップ手段に困っているエンジニア
- サーバーのディスク容量不足でローカル経由の転送が困難な方
- rclone を用いた高度な転送設定を学びたい方

## 前提知識
### rclone とは
rclone は「クラウドストレージ用の rsync」とも称されるオープンソースのコマンドラインツールです。Google Drive、Amazon S3、Dropbox、SFTP など、70種類以上の異なるストレージサービス間でファイルを同期、移動、コピーできます。強力なバッファリング機能や並列転送制御を備えており、ネットワーク経由のデータ管理におけるデファクトスタンダードのひとつです。

### 実践環境
本記事の内容は以下の構成に基づいています。
- クライアントマシン：M1 MacBook Pro（macOS）
- 転送元：さくらインターネット 共有サーバー（FreeBSD 11.2）
- 転送先：Google Drive

## 課題と背景
保守案件で直面した環境は、FreeBSD 11.2 という古いOSを搭載した共有サーバーでした。この環境では以下の問題が発生しました。

1. 標準のバックアップ機能 SnapUp が、ファイル数の多さとデータ容量によりタイムアウトする
2. 最新の rclone バイナリがOSのライブラリ依存関係により動作しない
3. サーバー内の空き容量が少なく、圧縮アーカイブを作成する余裕がない

これらの制約を打破するため、サーバー側に実行ファイルを置かず、クライアント側（Mac）から命令を出す構成を採用しました。

## 解決策のアーキテクチャ
クライアントマシンのメモリをバッファとして利用し、サーバーとクラウドストレージを直結します。

[Server (SFTP)] <--> [Mac (rclone)] <--> [Google Drive (API)]

この構成では、Mac のディスクを経由せずデータが流れるため、Mac 側の空き容量を気にする必要がありません。

## 実践セットアップ
まずは rclone の設定ファイルである rclone.conf を準備します。

### 接続先の設定
```ini
[sakura_server]
type = sftp
host = xxx.sakura.ne.jp
user = username
port = 22
key_file = ~/.ssh/id_rsa_sakura

[gdrive_backup]
type = drive
scope = drive
root_folder_id = 1WYLzwBZkRupq-nvdY5KyeYRz-9BjMbay
```
注釈として、root_folder_id は Google Drive のURL末尾にある文字列を指定することで、特定のディレクトリをルートとして扱えます。

### 実行コマンドの最適化
以下のコマンドは、共有サーバーの負荷を抑えつつ、小ファイルの転送効率を最大化するように設計されています。

```bash
caffeinate -i rclone copy sakura_server:www/target_path gdrive_backup:"backup_folder" \
  --transfers 10 \
  --checkers 20 \
  --fast-list \
  --buffer-size 32M \
  --drive-chunk-size 64M \
  --progress
```

各オプションの役割は以下の通りです。

- --transfers 10：同時並行で転送するファイル数。共有サーバーの接続制限を考慮した値です
- --checkers 20：差分チェックを行う並列数。CPU負荷を抑えつつリストアップを高速化します
- --fast-list：ディレクトリ一覧をメモリ上に保持し、APIリクエスト回数を削減します
- --buffer-size 32M：転送時のメモリバッファ量。ストリーミングの安定性を高めます

### コラム：長時間実行の安定化
大容量の転送には数時間を要するため、途中でシステムがスリープ状態に移行することを防ぐ必要があります。macOS では caffeinate コマンドを使用することで、プロセスの実行中にシステムアサーションを維持し、通信の切断を防止できます。

```bash
caffeinate -i
```

## 転送結果と知見
最終的な転送実績は以下の通りでした。

- 転送総量：約70 GiB
- ファイル数：58,730 件
- 所要時間：13 時間 21 分
- ステータス：Success（整合性確認済み）

当初は17GB程度と認識されていたデータが実体は約70GBに膨れ上がっていました。以前のバックアッププラグインが作成した巨大な一時ファイルが原因でしたが、今回の手法でそれらも含めてすべてクラウドへ確保できました。
