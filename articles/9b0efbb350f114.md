### はじめに

長年運用されてきたシステムには、時として現代のツールが通用しない「孤島」のような環境が存在します。本記事では、最新のツールが直接動作しない古いOSのサーバーから、手元のクライアントマシンをプロキシとして活用し、大容量データを安全かつ効率的にGoogle Driveへ同期させる手法を解説します。

### 対象者

* 古い共有サーバーのバックアップ手段に困っているエンジニア
* サーバーのディスク容量不足でローカル経由の転送が困難な方
* rcloneを用いた高度な転送設定を学びたい方

### 課題と背景

保守案件で直面した環境は、FreeBSD 11.2という古いOSを搭載したさくらインターネットの共有サーバーでした。この環境では以下の問題が発生しました。

1. 標準のバックアップ機能であるSnapUpが、ファイル数の多さとデータ容量によりタイムアウトする
2. 最新のrcloneバイナリがOSのライブラリ依存関係により動作しない
3. サーバー内の空き容量が少なく、圧縮アーカイブを作成する余裕がない

これらの制約を打破するため、サーバー側に実行ファイルを置かず、クライアント側（Mac）から命令を出す構成を採用しました。

### 解決策のアーキテクチャ

クライアントマシンのメモリをバッファとして利用し、サーバーとクラウドストレージを直結します。

[Server (SFTP)] <--> [Mac (rclone)] <--> [Google Drive (API)]

この構成では、Macのディスクを経由せずデータが流れるため、Mac側の空き容量を気にする必要がありません。

### 実践セットアップ

まず、rcloneの設定ファイルである rclone.conf を作成します。

#### 接続先の設定

[sakura_server]
type = sftp
host = xxx.sakura.ne.jp
user = username
port = 22
key_file = ~/.ssh/id_rsa_sakura

[gdrive_backup]
type = drive
scope = drive
root_folder_id = 1WYLzwBZkRupq-nvdY5KyeYRz-9BjMbay

（注釈：root_folder_idは、Google DriveのURL末尾にある文字列を指定することで、特定のディレクトリをルートとして扱えます）

#### 実行コマンドの最適化

以下のコマンドは、共有サーバーの負荷を抑えつつ、小ファイルの転送効率を最大化するように設計されています。

rclone copy sakura_server:www/target_path gdrive_backup:"backup_folder" \
  --transfers 10 \
  --checkers 20 \
  --fast-list \
  --buffer-size 32M \
  --drive-chunk-size 64M \
  --progress

各オプションの役割は以下の通りです。

* --transfers 10：同時並行で転送するファイル数。共有サーバーの接続制限を考慮した値です
* --checkers 20：差分チェックを行う並列数。CPU負荷を抑えつつリストアップを高速化します
* --fast-list：ディレクトリ一覧をメモリ上に保持し、APIリクエスト回数を削減します
* --buffer-size 32M：転送時のメモリバッファ量。ストリーミングの安定性を高めます

### 長時間実行の安定化措置

大容量の転送には数時間を要するため、途中でシステムがスリープ状態に移行することを防ぐ必要があります。macOSでは、caffeinateコマンドを使用することで、プロセスの実行中にシステムアサーションを維持し、通信の切断を防止できます。

caffeinate -i

### 結果

約1.4万ファイル、合計17GBのデータを約2時間で転送完了しました。一度ベースとなる同期が完了すれば、次回以降は sync コマンドにより差分のみを数分で更新可能です。この手法は、Windows環境であればPowerShellの実行ポリシー設定等で同様の管理が再現できます。

### 重要なキーワード

* rclone
* SFTP
* Google Drive API
* ストリーミング転送
* caffeinate
* データ整合性
* 差分同期
