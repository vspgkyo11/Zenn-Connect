## はじめに

2025年12月、React Server Components に関する重大な脆弱性があると告知されました。
本件は `CVE-2025-55182` として登録されており、React 19 系およびそれを利用する主要フレームワークに影響します。特に Next.js を利用しているプロジェクトでは、Next.js 15 から 16 までの全バージョンが影響範囲に含まれるため、迅速な対応が求められます。
本記事では、脆弱性の概要、影響範囲、そして具体的な対応方法をご紹介します。

## 対象者

* React Server Components を利用している方
* React 19 系および Next.js 15 ～ 16 を本番環境で運用している方
* 脆弱性対応をしなければならない方

## 結論

* `CVE-2025-55182` は RSC の評価過程に起因する重大な脆弱性であり、Next.js 15 と 16 に広く影響を与えます
* 既にパッチが提供されており、更新すれば影響は解消されます
* バージョンアップとログ確認を行うことで安全性を確保できます

## 脆弱性の概要

`CVE-2025-55182` は、React Server Components の設計と実装に起因する脆弱性です。
サーバー側で実行されるコンポーネントの境界が破られることで、本来公開されるべきでないデータや内部処理が露出する可能性が指摘されています。
この問題は単一のフレームワーク固有のものではなく、React 19 を基盤として Server Components を実装している複数のフレームワークに波及します。

:::message alert
**注意事項**

本脆弱性は、外部からの特定のリクエストによって情報漏洩する可能性があります。特に、公開中の本番環境が未修正のまま稼働していた場合、シークレットや内部 API の露出リスクの可能性があります。
:::

### Next.js への影響

Next.js では、React Server Components を中核機能として深く統合しています。
そのため、Next.js 側でも本脆弱性に対応する修正が行われ、別の CVE として `CVE-2025-66478` が割り当てられました。

以下はいずれも、脆弱性へのが対策されたバージョンです。

**影響を受けるバージョン**

* Next.js 15 系 全バージョン
* Next.js 16 系 全バージョン

**推奨される修正版へのアップデート**

* 15.0.5
* 15.1.9
* 15.2.6
* 15.3.6
* 15.4.8
* 15.5.7
* 16.0.7


### React 単体利用や他フレームワークの場合

Next.js 以外でも、React Server Components を採用しているフレームワークは影響を受けます。その場合は、React 自体を修正済みのバージョンへ更新する必要があります。

以下はいずれも Server Components に関するセキュリティ修正を含みます。

**推奨される React バージョン**

* 19.0.1
* 19.1.2
* 19.2.1

## 対応手順

対応手順の流れは以下の通りです。
 Step 1. 利用バージョンの確認
 Step 2: パッチ版への更新
 Step 3: ビルドとデプロイ
 Step 4. 本番デプロイ前の確認
 Step 5. 事後対応の検討

### Step 1. 利用バージョンの確認

まず `package.json` や `lock` ファイルで、以下を確認します。

* next のバージョン
* react および react-dom のバージョン

### Step 2: パッチ版への更新

```
$ pnpm add next@latest react@latest react-dom@latest
```

またはバージョンを固定する場合は次のように記述する。

```
$ pnpm add next@15.5.7 react@19.2.1 react-dom@19.2.1
```

### Step 3: ビルドとデプロイ

```
$ pnpm build
$ git commit -am "例）chore: security update RSC CVE-2025-66478"
$ git push
```

### Step 4. 本番デプロイ前の確認

以下の観点から動作確認をします。
特に、Server Actions やキャッシュ周りを使っている場合は重点的に確認するのがおすすめです。

* ビルドが通るか
* Server Components 周りの挙動に差分がないか
* 認証や権限周りの画面遷移が正しいか

### Step 5. 事後対応の検討

未修正の状態で公開されていた期間がある場合は、以下も検討対象です。

* シークレットの差し替え
* 認証トークンの再発行
* アクセスログの確認

本脆弱性は、直ちに機密情報が流出する形式ではありません。ただし、RSC 経路を悪用した想定外の挙動が存在した可能性は否定できないため、念のため次の確認をすることが望ましいです。

* 不審な RSC 関連リクエスト
* API キーに対する異常アクセス
* 想定外のサーバーログや不明な500エラーや警告
