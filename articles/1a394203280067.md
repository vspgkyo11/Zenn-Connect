## はじめに

さくらインターネットの共用サーバーで WordPress を MySQL 5.7 → 8.0 に移行しました。事前にデータベースのダンプファイルを静的解析するとアップグレードにより `sql_mode` の差異によるデータの欠落の可能性が出てきたため、データクレンジングを行いました。移行は成功しましたが、気になることがありました。

「さくら MySQL 8.0 アップグレード」記事の多くは「コントロールパネルのボタンを一度押したら完了した」で終わっています。クレンジングへの言及はほぼありません。それでも問題が起きていないのはなぜなのか。

ローカルDocker環境で再調査してみると、クレンジングなしでもアップグレードは成功しデータ欠落は発生しませんでした。本記事では、これらの背景と原因を掘り下げます。



## 前提：MySQL 5.7 → 8.0 で sql_mode が変わる

MySQL 5.7 → 8.0 の移行では `sql_mode` の厳格化が問題になると言われています。しかし **WordPress 経由の通常運用では、実はほぼ影響を受けません**。WordPress が接続のたびに自分で STRICT を除去しているからです。

MySQL 8.0 はデフォルトで `STRICT_TRANS_TABLES` が有効になるため、5.7 では警告のみで通過していた `0000-00-00 00:00:00`（無効な日付値）が、INSERT/UPDATE 時にエラーになりえます。しかし WordPress はその影響を自分で回避しています。詳細は後述します。

レンタルサーバーの MySQL 5.7 は `STRICT_TRANS_TABLES` が無効な設定が多く、WordPress やプラグインが `0000-00-00` を長年書き込み続けても誰も気づかない状況になりがちです。今回の本番 DB を解析したら `0000-00-00 00:00:00` が数千件見つかったのも、この構造的な背景によるものです。



## 1. WordPress は接続のたびに STRICT を自分で外している

WordPress コアの `wp-includes/class-wpdb.php`（バージョン 3.9.0 から存在）には、次の定義があります。

```php
protected $incompatible_modes = array(
    'NO_ZERO_DATE',
    'ONLY_FULL_GROUP_BY',
    'STRICT_TRANS_TABLES',
    'STRICT_ALL_TABLES',
    'TRADITIONAL',
    'ANSI',
);
```

WordPress は DB 接続時に `SET SESSION sql_mode = ...` を発行し、このリストに含まれるモードをセッションから除去します。

MySQL 8.0 のデフォルトでは `STRICT_TRANS_TABLES` が有効ですが、WordPress 経由で接続した瞬間に取り除かれます。実際に WordPress の DB 接続中に `sql_mode` を確認すると：

```
NO_ZERO_IN_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_ENGINE_SUBSTITUTION
```

`STRICT_TRANS_TABLES` も `NO_ZERO_DATE` も消えています。この状態では `0000-00-00 00:00:00` を INSERT しても MySQL 8.0 はエラーにしません。

WordPress が `wp-includes/post.php` で `post_date_gmt = '0000-00-00 00:00:00'` を意図的に書き込むコードが今も存在するのは、この自衛機構があるからです。MySQL 8.0 でも、WordPress の通常運用は翌日から壊れません。



## 2. mysqldump も STRICT を回避する

さくらの自動アップグレード機能は内部的に mysqldump でダンプを取り、MySQL 8.0 に再インポートします。

phpMyAdmin でエクスポートした `.sql` ファイルの冒頭には、必ず次のような行が含まれています。

```sql
SET SQL_MODE = "NO_AUTO_VALUE_ON_ZERO";
SET time_zone = "+00:00";
```

この `SET SQL_MODE` によってインポート中の sql_mode が上書きされるため、`0000-00-00` を含む INSERT 文がそのまま通ります。「ボタン一発」でアップグレードが成功していた理由はここにあります。



## 3. アクセス経路ごとの整理

| アクセス経路                    | sql_mode の扱い                       | `0000-00-00` への影響       |
| ------------------------------- | ------------------------------------- | --------------------------- |
| WordPress（wpdb 経由）          | 接続時に STRICT / NO_ZERO_DATE を除去 | INSERT/UPDATE 成功          |
| phpMyAdmin                      | サーバー設定をそのまま使用            | SELECT でも ERROR 1525      |
| mysqldump → インポート          | dump 冒頭の `SET SQL_MODE` で除去     | インポート成功              |
| mysql コマンド直接 / 一部ツール | サーバー設定をそのまま使用            | INSERT でエラーになる可能性 |

WordPress 経由の通常運用は保護されています。問題になるのは「WordPress を通らない」文脈だけです。



## 4. では、クレンジングは何のためだったのか

「WordPress も守っている、ダンプも守っている、なら調査もクレンジングも無駄だったのか」というと、そうではありません。

**phpMyAdmin で直接 SQL を実行するとき**

phpMyAdmin の SQL タブから直接クエリを実行する場合、sql_mode はサーバー設定がそのまま使われます。MySQL 8.0 環境で `0000-00-00` を比較値に使うだけでエラーになります。

```sql
SELECT * FROM wp_posts WHERE post_date_gmt = '0000-00-00 00:00:00';
-- ERROR 1525 (HY000): Incorrect DATETIME value: '0000-00-00 00:00:00'
```

DB に不正値が残っている限り、phpMyAdmin での直接操作は地雷だらけです。

**サードパーティツールでバックアップ・リストアするとき**

WordPress とダンプが用意している「回避」は、それぞれの文脈でしか機能しません。別のバックアッププラグインや直接の `mysql` コマンドでインポートした場合、STRICT が有効なまま `0000-00-00` を挿入しようとしてエラーになる可能性があります。

**DB 自体が MySQL 8.0 の標準に準拠していない状態が続く**

WordPress が回避してくれていても、DB の中に `0000-00-00` が残り続けることは変わりません。将来の移行や外部ツール導入のどこかで予期しない問題として浮上するリスクが残ります。

> **結論**：クレンジングは「WordPress を守るため」ではありませんでした。WordPress は自分で守る仕組みを持っています。クレンジングとは、WordPress に依存せず、**DB 自体を MySQL 8.0 の仕様に準拠させる作業**でした。



## 5. 「なぜ成功するのか」を理解することの意義

「ボタン一発で大丈夫」は正しいです。ただしその理由は「MySQL 8.0 が緩い」からではなく、**WordPress が賢く立ち回っているから**です。

その構造を理解した上でクレンジングするのと、何も知らずにボタンを押すのでは、同じ結果でも意味が違います。phpMyAdmin での直接操作やサードパーティツール導入のときに、どこでエラーが出てどこでは出ないかを判断できるようになります。



## まとめ

| 問い                                          | 答え                                                                               |
| --------------------------------------------- | ---------------------------------------------------------------------------------- |
| クレンジングしなくても WordPress は動いたか？ | **動いた**（WordPress が STRICT を外しているため）                                 |
| ダンプのインポートは成功したか？              | **成功した**（dump 冒頭の `SET SQL_MODE` が機能するため）                          |
| ではクレンジングは不要だったか？              | **DB の健全性のためには必要**（phpMyAdmin 操作、外部ツール、将来の移行に影響する） |
| 事前調査に意味はあったか？                    | **あった**。「なぜ壊れないのか」の仕組みがわかったこと自体が成果                   |


## おわりに
レンタルサーバーのアップグレードは、「ボタンを押す」だけで完了することが多くなっています。なぜ成功するのかを知らずにいると、予期しないタイミングで——phpMyAdmin でエラーが出た、バックアップツールが動かなくなった——原因の見当もつかない、という状況に陥ることがあります。今回、MySQL アップグレードに関する調査でしたが、結果として WordPress の内部構造について思わぬ発見がありました。その気づきが、今回の一番の収穫だったかもしれません。このような裏側の仕組みを知っておくことが、問題が起きたときの診断力に直結すると思います。本記事がその一助になれば幸いです。
