## はじめに

Zenn への投稿を続けていくと、`articles/` の中に大量の Markdown ファイルが溜まっていきます。私のリポジトリもすでに75本を超え、ファイル名を見ても内容がまったく分からない状態になっていました。

```
articles/efd31b50df3172.md
articles/1a394203280067.md
articles/e1e6b072e89c27.md
```

「不規則な文字列の羅列でリネームできないなら、いっそ全部書き換えてしまおうか」と考えたのですが、調べてみるとこれは**触ってはいけない領域**でした。本記事では、なぜファイル名を変えてはいけないのか、その代わりにどう整理すれば見通しがよくなるのかをまとめます。

## 対象者

* Zenn の記事数が増えて `articles/` の中が把握しづらくなってきた方
* ランダムなファイル名を「見やすくリネームしたい」と思ったことがある方
* 個人運用の Zenn リポジトリに何らかの管理ルールを導入したい方

## 結論

* **公開済み記事のファイル名は絶対にリネームしない**
* 見やすさは「索引ファイル」を別に用意して解決する
* 命名ルールは**これから作る記事にだけ**適用する

## なぜファイル名を変えてはいけないのか

Zenn では、**ファイル名（スラッグ）がそのまま記事の公開URLになります**。

```
articles/efd31b50df3172.md
→ https://zenn.dev/{username}/articles/efd31b50df3172
```

そのため、公開済みの記事のファイル名を変更すると、Zenn 側からは「旧記事が削除されて、別の記事が新規投稿された」ように見えてしまいます。具体的には次のようなことが起こります。

* 記事に付いていた「いいね」やコメントが失われる
* 検索エンジンに評価されていたURLが 404 になる
* SNSでシェアされていたリンクや、他サイトからの被リンクが切れる

:::message
あのランダムな文字列は「見にくいファイル名」ではなく、記事に紐づく**永続的なID**だと捉えるのが正確です。見た目のためにIDを書き換えることはできません。
:::

75本すべてがすでに公開済みだったので、私のケースでは「ファイル名のリネームは1本もしない」という判断に落ち着きました。

## 見やすさは「索引レイヤー」で解決する

ファイル名を変えられない以上、人間が記事を探すための仕組みを別に用意するのが現実的な解決策です。私はリポジトリに次の2つを用意しました。

* `docs/article_index.md` … 全記事のタイトル・トピック・概要を一覧化した索引
* `scripts/generate_index.py` … `articles/` の Markdown を走査して索引を自動生成するスクリプト

```bash
python3 scripts/generate_index.py
# => Generated index at docs/article_index.md (75 articles)
```

フロントマターの `title` や `topics` を集計するだけの単純なスクリプトですが、これがあるだけで「ファイル名は機械が扱うID、索引ファイルが人間の目次」という役割分担がはっきりします。記事を追加したら索引を再生成する、というルールさえ守れば、ファイル名の不規則さはほとんど気にならなくなりました。

:::message
Homebrew管理のPythonだと `pip install` がシステム全体にブロックされることがあります。その場合は `python3 -m venv` で仮想環境を作ってからライブラリを入れると安全です。
:::

```bash
python3 -m venv .venv-scripts
.venv-scripts/bin/pip install python-frontmatter
.venv-scripts/bin/python3 scripts/generate_index.py
```

## これから作る記事には命名ルールを適用する

過去の記事はどうにもなりませんが、**新規記事のスラッグは自分で指定できます**。Zenn CLI の `--slug` オプションを使えば、ランダム文字列を避けられます。

```bash
npx zenn new:article --slug laravel-session-separation --title "タイトル" --type tech --emoji 🔐
```

スラッグには次の制約があるので、決め切ってから作成するのがおすすめです。

* 使える文字は半角英小文字・数字・ハイフン・アンダースコアのみ
* 長さは12〜50文字
* **公開後は変更できない**（今回学んだ制約そのもの）

私は `<カテゴリ>-<トピック>` という形式に決めました。

| 例 | 意図 |
| --- | --- |
| `laravel-session-separation` | Laravel × セッション分離 |
| `nextjs16-middleware-to-proxy` | Next.js 16 の middleware→proxy 移行 |

## おわりに

「ファイル名が気に入らないから直したい」という気持ちは自然ですが、Zenn の場合はファイル名がそのままURLという仕組み上、公開済み記事には手を出さないほうが安全です。整理したくなったら、ファイル名そのものではなく「索引」と「これからの命名ルール」の2つで解決する、という考え方が個人的にはしっくりきました。

同じように記事が増えて迷子になっている方の参考になれば幸いです。
