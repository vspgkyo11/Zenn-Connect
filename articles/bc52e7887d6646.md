# はじめに

Prisma 7 へのメジャーアップデートに伴い、Prisma Client の生成場所（output）をデフォルトの node_modules からプロジェクト内のディレクトリ（例: src/generated/prisma）へ変更する Custom Output を導入した際、思わぬビルド失敗に遭遇することがあります。

特筆すべきは、Vercel などの CI 環境において、具体的なエラーログが出ないまま Exit code: 1 で終了してしまう「サイレントなビルド死」が起きる点です。本記事では、Prisma 7 の公式ドキュメントの根拠に基づき、この現象の技術的な背景と、確実にビルドを通すための修正ポイントを解説します。

# 対象者

- Prisma 7 への移行を検討している、または移行途中の開発者
- Next.js プロジェクトで Custom Output を導入し、型安全な環境を構築したい方
- CI で原因不明のビルドエラーに悩まされている方

# 発生する問題：Vercel でのサイレントな失敗

Next.js 16 と Prisma 7 の環境でビルドを実行した際、以下のような症状が発生し、デプロイが即座に失敗することがあります。

- 症状
Command "pnpm vercel-build" exited with 1 とだけ表示される。
- 特徴
通常表示されるはずの TypeScript の型エラー詳細や、Next.js のビルド進捗ログがほとんど出力されず、プロセスの極めて初期段階で終了する。

これは本格的なバンドル処理や静的生成が始まる直前の「型チェック」段階で、プロジェクト内のいずれかのファイルが致命的な型解決エラーを起こしていることを示しています。

# 技術的根拠：Prisma 7 の仕様変更

なぜこのエラーが起きるのか。その答えは Prisma 7 の公式ドキュメントの Custom output location セクションに明記されています。

「カスタム出力場所を使用する場合、生成された Prisma Client は node_modules/.prisma/client には配置されません。代わりにカスタム場所からインポートする必要があります」

デフォルト設定では @prisma/client パッケージが node_modules/.prisma/client へのエイリアスとして機能しますが、独自の出力先を指定した時点でこの自動リンクが失われます。つまり、標準の import { PrismaClient } from "@prisma/client" という記述は、型定義も実体も持たない空の参照へと変わるのです。

# 真の原因：取り残された外部スクリプト

多くのプロジェクトでは、src/ 配下の主要なアプリケーションコードは新しいインポートパスへと一括置換されます。しかし、以下の場所にあるファイルが盲点となります。

- プロジェクトルートの scripts/ フォルダ内にある運用ツール
- データベースの Seed スクリプト (prisma/seed.ts など)
- 独自のマイグレーション用スクリプト

これらが tsconfig.json のスキャン対象に含まれている場合、Next.js のビルド内で走る型チェックが「古いインポート（from "@prisma/client"）」を見つけ出し、ビルドプロセスを強制終了させます。最新のビルド環境ではログのバッファがフラッシュされる前にプロセスが終了されることがあり、結果として Vercel 上では原因不明の失敗として記録されることになります。

# 解決策：2つのステップ

この問題を解消し、安定したビルドを実現するためには、以下の2段階の修正が必要です。

### 1. インポートパスの物理パス修正

src/ ディレクトリ外にあるスクリプト類も、生成された Prisma Client を相対パスで直接参照するように修正します。

以前（Custom Output 導入後は型が見つからない）
import { PrismaClient } from '@prisma/client'

以降（生成先を直接指定）
import { PrismaClient } from '../src/generated/prisma'

### 2. tsconfig.json での保護

運用スクリプトの型不整合によって本番ビルドが止まるのを防ぐため、またビルド時間を短縮するために、exclude に scripts ディレクトリを明示的に追加します。

```json
{
  "compilerOptions": { ... },
  "exclude": [
    "node_modules",
    "scripts"
  ]
}
```

# コラム：なぜこのミスは見逃されたのか

今回のトラブルの背景には、AIによる局所的な最適化と、エンジニアの全体像把握の漏れという、現代の開発現場で起こりがちな構図が隠れています。

1. AIのコンテキストの限界
AIにエラーを相談すると、提示されたコードの範囲内では完璧な修正案が出されます。しかし、プロジェクトルートにある実務的なスクリプトの存在まで即座に考慮されることは稀です。エンジニアがAIの提案を信頼しすぎることで、特定のディレクトリは正常だがプロジェクト全体としては不整合である、という中途半端な状態が維持されてしまいます。

2. node_modules という魔法への依存
これまでの Prisma は、コマンド一つで自動的に型が解決される便利な仕組みを提供していました。多くのエンジニアがその内部仕様を意識せずに恩恵を受けていたため、Prisma 7 でその仕組みが大きく変更された際、自分たちのコードがどこに依存しているのかを正確に見積もることが難しくなっています。

3. スクリプトフォルダの優先順位
アプリケーションの核となるコードは常に注視されますが、運用スクリプトは道具箱のような扱いになりがちです。ローカルの主要な開発フローでは実行されないためエラーに気づかず、ビルド時に初めて致命的な問題として浮上することになります。

最新のツールを使いこなしつつも、常にシステムの全体像を捉え、基本に立ち返って整合性を確認する重要性を再認識させられる事例と言えます。

# おわりに

Prisma 7 における Custom Output への移行は、ライブラリが node_modules を汚染しないクリーンな構成を可能にしますが、プロジェクト内のすべての TS ファイルが影響を受けるというトレードオフがあります。

公式の指針どおり、@prisma/client への依存を断ち切り、正しい参照先へ向け直すことが、安定したビルドへの近道です。こうした細かな修正の積み重ねが、最終的には開発体験の向上に繋がります。この記事が、同じ悩みを持つ開発者のみなさんの助けになれば幸いです。
